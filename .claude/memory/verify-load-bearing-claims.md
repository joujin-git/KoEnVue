어떤 주장이 (1) **정량적**이고 (2) 그 위에 권고·결정이 **통째로 얹혀 있으면**, 출처가 서브에이전트든 코드 주석이든 메인 세션이 **조건을 바꿔 직접 재현**한다. [[tool-limit-verify-first]] 가 "안 된다"는 부정 주장을 실험으로 뒤집는 것이라면, 이쪽은 "그렇다"는 긍정 주장을 실험으로 떠받치는 쪽이다.

**Why:** 2026-07-22 커서 인디 지연 조사에서 워크플로우 에이전트가 "`SetTimer(16)` 은 실제 ~31ms 로 배달된다"는 실측을 냈고, 그게 1순위 권고의 **유일한** 근거였다. 메인 세션이 벤치를 직접 짜 재현하니 (a) 결론은 맞았고 (b) 1차 측정에서 자체 버그를 잡아냈다 — `SetTimer(NULL,…)` 은 `nIDEvent` 를 무시하고 새 ID 를 발급해 `KillTimer(NULL, 원래ID)` 가 실패, 타이머가 누적되며 이후 전 케이스가 오염됐다(median 0.05ms 라는 물리적으로 불가능한 값이 징후). 동시에 **코드 주석 `16ms ≈ 60fps` 가 여러 세션 동안 사실로 통용되며 이전 세션 문서의 "폴링은 병목 아님" 결론까지 전파**돼 있었다 — 아무도 재보지 않아서 생긴 오류였다.

**How to apply:** 재현은 **서브에이전트와 다른 조건**으로 — 더 실제 환경에 가깝게 잡으면 일치 시 신뢰도가 크게 오른다(위 사례: 에이전트는 스레드 타이머, 메인은 앱이 실제 쓰는 창 타이머). 결과에 물리적으로 불가능한 값이 섞이면 대상이 아니라 **측정 코드를 먼저 의심**한다. 재현이 비싸면(빌드·실기 필요) 최소한 그 주장이 load-bearing 이라는 사실과 미검증 상태를 결과물에 명시한다.
