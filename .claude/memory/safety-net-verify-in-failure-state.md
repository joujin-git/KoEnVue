hook·fallback·복구 로직 같은 **안전망은 정상 상태에서의 성공 관측으로 검증했다고 착각하기 쉽다**. 안전망이 존재하는 이유인 **실패 상태를 인위적으로 만들어 end-to-end 로 발화**시켜야 검증이다. 함수 단위 호출 성공은 배선 검증이 아니다.

**Why:** `Sync-Memory`(2026-06-08 신설, C: 복원 대비 메모리 복구 hook)는 검증이 "C: 디렉토리가 **살아있는** 부분 복원" 상태에서만 이뤄졌다(`restored=5`, C:=E: 해시 일치). 정작 존재 이유인 **완전 소실**(프로젝트 디렉토리째 새로 생성 → `memory` 부재)에서는 `Get-AutoMemoryDir` 이 `Test-Path` 실패로 `$null` → `Sync-Memory` 가 즉시 `return` → **무동작**. E: 쪽은 없으면 `New-Item` 으로 만들면서 C: 쪽만 포기하는 비대칭이었다. 2026-07-22 에 실제 발생해 **6주간 메모리 recall 0건**이었는데, 알림 조건이 `absorbed>0 -or restored>0` 이라 실패가 **침묵**했다 — 무동작과 "정상이라 변화 없음"이 구분되지 않아 아무도 몰랐다. 2026-06-09 세션은 이 상태를 보고도 "C: 가 비었으니 실제 활성 store 는 E:" 라고 정반대로 해석했다(E: 에 파일이 쌓인 건 hook 이 백업한 결과지 런타임이 쓴 증거가 아님).

**How to apply:** 안전망을 만들거나 고치면 ① 발동 조건인 **실패 상태를 실제로 만들어**(디렉토리 이동·깨진 입력 주입·프로세스 강제 종료 등) 실제 실행 경로로 돌린다 — 프로브는 직후 흔적 0 원복. ② **무동작과 정상 무변화가 로그/알림에서 구분되게** 한다. 침묵하는 실패 경로는 없는 것과 같다. 관련: [[tool-limit-verify-first]](못 한다 단정 전 실험 — 이쪽은 "된다고 확인한 것도 실패 상태에서 재확인"), [[os-dependent-accept]].
