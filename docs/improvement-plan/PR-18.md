### 1.2 ARM64 NativeAOT volatility 비대칭 회복 — hwnd 3종

[Program.cs:36-46](../../Program.cs#L36) 의 전역 상태에서 비대칭 발견:

| 필드 | volatile | cross-thread read | cross-thread write |
|------|---------|------------------|------------------|
| `_config` (L44) | ✅ | 감지 스레드 | 메인 스레드 |
| `_lastImeState` (L45) | ✅ | 메인 스레드 | 감지 스레드 |
| `_indicatorVisible` (L46) | ✅ | 감지 스레드 | 메인 스레드 |
| `_hwndMain` (L36) | ❌ | 감지 스레드 (8× `PostMessageW(_hwndMain, ...)` L1265-1477) | 메인 스레드 (`CreateMainWindow` L223) |
| `_hwndOverlay` (L37) | ❌ | 감지 스레드 (`IsKoenvueWindow` L1220) | 메인 스레드 (L252) |
| `_hwndCursorOverlay` (L41) | ❌ | 감지 스레드 (L1220) | 메인 스레드 (L843, L865, L887) |

x64 의 TSO (Total Store Order) 메모리 모델 + 단일 init-then-read 패턴 덕에 현재 회귀 0. PR-19 의 `_detectionThread?.Join(500)` 가 ProcessExit 시점 race window 를 좁혔으나, **ARM64 weak memory model + NativeAOT codegen** 조합에서는 init 직후 감지 스레드가 stale `IntPtr.Zero` 를 보고 silent no-op 할 가능성이 남는다. cost 0 (volatile keyword 6 글자) 의 방어 패치.

## 3. 비범위 (Out of Scope) — premultiply 후처리 의미 차이

[LayeredOverlayBase.cs:715-740](../../Core/Windowing/LayeredOverlayBase.cs#L715) `ApplyPremultipliedAlpha` 와 [LayeredCursorBase.cs:264-298](../../Core/Windowing/LayeredCursorBase.cs#L264) 의 동명 메서드는 **a==0 가드의 의미가 반대**이다:

| 엔진 | a==0 && RGB!=0 픽셀 처리 | 의도 |
|------|------------------------|------|
| overlay (LayeredOverlayBase) | `ptr[offset + 3] = 255` (alpha 255 복구) | GDI DrawTextW AA 엣지 보존 |
| cursor (LayeredCursorBase) | `ptr[offset] = ptr[offset+1] = ptr[offset+2] = 0` (RGB 제로화) | ShadeDib 의 round-down 부산물 (avgAlpha × 255 < 0.5) 잡티 제거 |

[LayeredCursorBase.cs:279-282](../../Core/Windowing/LayeredCursorBase.cs#L279) 의 인라인 코멘트가 이 의미 차이를 이미 명시 — overlay 의 a=255 복구가 cursor 엔진에 적용되면 fully-opaque 외곽 점이 잡티가 된다. 의미 다른 채로 두는 것이 올바른 설계. PR-18 은 이 메서드를 공유화하지 **않는다**.

## 6. 사이드 이펙트 / 위험

### 위험 1: DibSectionFactory 의 caller-side 상태 갱신 누락

EnsureDib 의 `_dibFailureLogged` / `_ppvBits` / `_currentBitmap` / `_currentWidth` / `_currentHeight` / `_lastRenderedStyle = null` 갱신은 **caller 책임**. 한 줄이라도 누락하면:
- `_ppvBits` 미갱신 → premultiply 가 stale 버퍼에 작업 → 직전 frame 표시
- `_lastRenderedStyle = null` 누락 → 동일 스타일로 인식해 DIB 재생성 후 그리기 스킵 → 빈 frame
- `_currentBitmap?.Dispose()` 누락 → SafeBitmapHandle 누수 → 장시간 실행 시 GDI handle 고갈

**완화**: commit 1 (overlay) + commit 2 (cursor) 의 PR 본문에 caller-side 갱신 6 항목 체크리스트. reviewer 가 diff 양면 비교.

### 위험 2: LayeredWindowBlit 의 BLENDFUNCTION 인라인 vs 캡슐화 결정

현재 두 엔진의 BLENDFUNCTION 은 100% 동일 (AC_SRC_OVER + flags 0 + alpha + AC_SRC_ALPHA). 그러나 PR-18 이후 한 엔진에서 BlendFlags 가 다른 값이 필요해지면 (예: cursor 의 hover state 가 BlendFlags=AC_USE_HIGHLIGHT 등) LayeredWindowBlit 의 캡슐화가 오히려 장애물.

**완화**: BLENDFUNCTION 은 **helper 안으로 흡수** (caller 는 alpha 만 전달) — 현재 사용 패턴 100% 일치라 YAGNI. 미래에 분기 필요 시 helper 시그니처를 BLENDFUNCTION 직접 받는 오버로드 추가하는 패턴 유지.

### 위험 3: volatile IntPtr 의 AOT codegen 경고

volatile IntPtr 는 .NET 5+ 에서 합법이지만, NativeAOT ILC codegen 이 ARM64 타깃에서 acquire/release barrier 를 옳게 emit 하는지는 **PR-19 의 x64 smoke 로는 검증 불가**. 본 PR 는 x64 빌드만 검증 — ARM64 codegen 회귀는 후속 PR / 실 사용자 보고로 발견.

**완화**: commit 5 의 변경은 **isolated** — 다른 4 commit 과 무관해 단독 revert 가능. dev-notes 에 "ARM64 evidence 부재, x64 smoke 무회귀" 정직 명시.

### 위험 4: P4 / P6 invariant 위반 잠재

신규 Core 모듈이 App 도메인 어휘 (label / IME / theme) 를 누출하지 않도록 검증. PR-08 의 `git grep -E "Hangul|English|NonKorean" Core/` 가드 0 매치 유지.

**완화**: §5.2 grep 가드에 P6 (KoEnVue\.App Core/ 0 매치) + PR-08 P4 (Hangul|English|NonKorean Core/ 0 매치) 두 항목 명시.

### 위험 5: AOT 사이즈 +5 KB 초과

DibSectionFactory + LayeredWindowBlit 두 신규 클래스가 +5 KB 안에 들어가야 함. 예측 ±0 ~ +2 KB (static class + 단일 메서드라 vtable 비용 0, 호출자 측 코드 감소 분과 상쇄).

**완화**: 각 commit 후 `Get-FileHash KoEnVue.exe` + 사이즈 측정. +3 KB 초과 시 reviewer 호출해 root cause 검토.

## 7. 롤백 절차

- 5 commit 의 각 commit 은 **독립 revert 가능**. dependency 는 commit 1 → 2 (DibSectionFactory signature) / commit 3 → 4 (LayeredWindowBlit signature) 만 — 나머지는 자유.
- 한 시나리오만 회귀 시 (예: cursor 잡티 ②) → commit 2 + 4 만 revert, overlay 측 1 + 3 + 5 는 유지 가능
- 전 회귀 시 (예: 모든 frame stuck) → 5 commit 전부 revert
- 데이터 영향 없음 — config.json / log 형식 변경 0
