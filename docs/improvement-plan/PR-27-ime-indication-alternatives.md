# PR-27: IME 상태 표시 방식 대안 탐색 — 커서 교체 기각, 캐럿 트윈 후보 도출

> 상태: **탐색 완료 / 미착수** — 2026-07-22. 코드 변경 0. 채택 결정은 사용자 검토 중.
>
> 목적: "커서 모양을 바꾸면 되지 않나?" 류의 아이디어가 **왜 안 되는지** 근거를 보존한다. 재조사 비용이 크다(본 탐색에 44 에이전트 / 3.35M 토큰 소요).

## 동기

한/영 **감지는 이미 거의 정확**하다. 남은 문제는 **알아낸 상태를 사용자에게 어떻게 알릴 것인가** 다. 현재 방식은 두 가지뿐:

1. **메인 인디케이터** — 포그라운드 창 근처 플로팅 라벨(한/En/EN)
2. **커서 인디케이터** — 마우스 포인터 주위 동심원 그래픽

사용자 질문 그대로: *"이 둘 말고 더 좋은 방법이 있을까? 예를 들어 한/영 상태에 따라 마우스 커서 자체의 모양을 바꿔버린다거나…"*

## 방법

워크플로우 `ime-indication-alternatives`(즉석 작성) — 5개 렌즈 병렬 발굴 → **대안별 Win32 API 실존 검증**(Microsoft Learn 공식 문서 확인, 지어낸 API 는 `INFEASIBLE`) → 사용자 가치·거슬림·시스템 오염 위험 심사.

**44 에이전트 / 실패 0 / 발굴 20건 → 기술적 탈락 1건 → 심사 19건.**

렌즈: 커서 자체 변형 · 입력 지점 앵커 · 주변시 · 비시각 · OS 통합.

## 결론 ① — 커서 자체 교체는 막다른 길 (4건 전부 REJECT, 위험 8~9/10)

### 용어 정정 — 마우스 포인터 ≠ 캐럿

혼동하기 쉬우나 **별개의 물건이고 화면에 동시에, 서로 다른 자리에 존재**한다.

| | 마우스 포인터 | 캐럿 |
|---|---|---|
| 만드는 주체 | 마우스 | 키보드 |
| 위치 | 마우스를 마지막에 둔 자리 | 글자가 실제로 찍히는 자리 |
| 텍스트 위에서 모양 | I-beam(⌶)으로 바뀜 | 깜빡이는 세로 막대(고정) |

마우스를 텍스트 위에 올리면 포인터가 I-beam 모양이 되지만 **그게 캐럿이 되는 것은 아니다.**

### 기각 사유 A — 타이핑 중 마우스 포인터가 숨겨진다 (설계 전제 붕괴)

실측: `SystemParametersInfo(SPI_GETMOUSEVANISH)` = **1 (Windows 기본값 ON)**

Windows 는 키 입력이 시작되면 마우스 포인터를 감춘다. 그런데 **한/영을 잘못 알아 문장을 날리는 사고는 타이핑하는 도중에 일어난다** — 신호가 가장 필요한 순간에 정확히 사라진다. API 문제가 아니라 설계 전제의 문제라 코드로 해결 불가.

**더 근본적으로**: 숨겨지지 않더라도 포인터는 마우스를 마지막에 둔 자리에 있다. 키보드로 입력할 때 사용자 시선은 캐럿에 있고 마우스는 화면 어딘가에 방치돼 있다. **이는 현재 커서 인디케이터가 이미 겪고 있는 한계와 동일한 문제다.**

### 기각 사유 B — 강제 종료 시 원복 불가 (시스템 오염)

`SetSystemCursor` 는 **세션 전역** 상태다. 작업 관리자의 "작업 끝내기"(`TerminateProcess`)는 `WM_CLOSE`/`WM_ENDSESSION`/`ProcessExit` 어느 것도 실행시키지 않으므로, **로그오프할 때까지 시스템 전체 커서가 변경된 채 남는다.** 재실행 시 자가치유(`SPI_SETCURSORS`)는 가능하나 사용자가 다시 켜야 하고, 그 `SPI_SETCURSORS` 자체가 **접근성 "커서 크기" 설정을 최소값으로 리셋하는 알려진 버그**를 유발한다.

현재 두 방식은 **자기 소유 레이어드 윈도우뿐이라 프로세스와 함께 사라진다** — 질적 차이다.

### 부가 기술 장벽

- `OCR_IBEAM` 은 `hbmColor=NULL` 인 **모노크롬 AND/XOR 커서** — 틴트할 색 자체가 없다. `DrawIconEx` 합성 경로가 무효이며 `GetDIBits` 로 1bpp 를 직접 읽어 32bpp BGRA 를 자체 합성해야 한다.
- 대기(`OCR_WAIT`)·작업중(`OCR_APPSTARTING`)은 `.ani` 애니메이션 커서 — 정적 합성 불가.
- `SetSystemCursor` 는 `SM_CXCURSOR` 크기에 고정 — "실루엣 바깥 1~2px 외곽선" 류 설계는 성립하지 않는다.

## 결론 ② — 1위 후보: **캐럿 트윈** (PROTOTYPE, 7/10)

캐럿(`rcCaret`)과 **동일한 폭·높이의 색 막대를 실제 캐럿에 밀착**시킨다. 라벨도 배지도 아니라서 사용자는 *"배지가 하나 더 떴다"* 가 아니라 **"내 텍스트 커서가 빨강 = 한글이 됐다"** 로 읽는다.

| 축 | 점수 | 근거 |
|---|---|---|
| 사용자 가치 | **8/10** | 심사 기준 *"입력하려는 순간, 시선이 있는 곳"* 을 만족하는 **유일한 후보**. 현재 라벨은 창 모서리(기본 BottomRight `-69,-58`), 커서 인디는 마우스 포인터인데 타이핑 중 포인터는 캐럿과 무관한 곳에 있다 — **시선 지점 표시는 현재 미충족 영역** |
| 거슬림 | 4/10 | 신규 화면 요소 0 — 캐럿이 이미 점유한 픽셀만 사용 |
| **위험** | **2/10** | **전역 상태 변경 0.** 자기 레이어드 윈도우 + `UnhookWinEvent` 뿐이라 강제 종료돼도 커널이 정리 — 원복 로직 자체가 불필요 |

### 구현 경로

- **1차**: `GetGUIThreadInfo` → `hwndCaret` / `rcCaret`(클라이언트 좌표) / `flags & GUI_CARETBLINKING` → `ClientToScreen`. **이미 [Program.cs:1330](../../Program.cs) 이 같은 구조체를 `hwndFocus` 용으로 호출 중**이라 신규 P/Invoke 는 `ClientToScreen` 정도.
- **추적**: 캐럿은 HWND 가 없어 `SetWinEventHook(EVENT_OBJECT_LOCATIONCHANGE)` 콜백에서 `idObject == OBJID_CARET` 필터가 **유일한 수단**(MSAA 문서). 80ms 폴링으로는 착시가 성립하지 않으므로 **훅은 최적화가 아니라 필수 요건**.
- **렌더**: [`Core/Windowing/LayeredCursorBase`](../../Core/Windowing/LayeredCursorBase.cs) 가 이미 `Show(x, y)` 를 받는 범용 클릭투과 추종 엔진이고 모니터별 DPI 재계산·이종 DPI 이동 좌표 race 까지 해결돼 있다. **렌더 콜백만 동심원 → 막대로 교체**하면 된다.
- **2차 폴백(보류 권장)**: `oleacc.dll` `AccessibleObjectFromWindow(OBJID_CARET)` → `IAccessible::accLocation`. 단 `IAccessible` 은 `IDispatch` 파생이라 소스생성 COM(IUnknown 베이스만 지원)으로 안 되고 슬롯 22개를 수동 선언해야 하며, **크로스프로세스 `IAccessible` 은 블로킹 COM RPC** 다 — 이 저장소가 같은 이유로 `SendMessage` 를 금지하고 `SendMessageTimeoutW` 만 쓰는([App/Detector/ImeStatus.cs](../../App/Detector/ImeStatus.cs)) 것과 정확히 같은 위험.

### ⚠ 치명적 제품 결함 — 침묵 실패가 거짓 신호가 된다

사용자가 "빨강 = 한글"로 학습하면, **캐럿을 못 잡는 앱에서 '막대 없음'을 '영문'으로 읽는다.** 표시기가 없어서 모르는 게 아니라, **없다는 사실이 틀린 상태를 적극적으로 주장한다** — 제품이 막으려는 바로 그 오류를 생산한다.

필수 설계 결정 2가지(타협 불가):
1. **한/영 양쪽 모두 각각 색을 칠해** '부재'가 상태가 되지 않게 한다.
2. 캐럿 미검출 시 **기존 라벨로 자동 폴백**. 설정 항목이 이미 98개([AppConfig.cs](../../App/Models/AppConfig.cs))이므로 사용자 선택으로 떠넘기지 말고 자동 하나로 고정.

### PROTOTYPE 인 이유

이 안의 가치 전부가 *"내 캐럿이 색을 바꿨다"* 는 **착시 성립 여부**에 걸려 있는데, 그건 분석으로 답할 수 없는 지각의 문제다. edit control `rcCaret` 폭 오차(MS 문서 명시) · 추적 지연 · 실제 캐럿 깜빡임과의 맥놀이 셋 중 하나만 걸려도 "색 바뀐 내 캐럿"이 아니라 **"캐럿이 두 개"** 로 읽히고, 그건 모서리 라벨보다 더 피로하다.

**판정 기준은 딱 하나** — 메모장/VS Code/Chrome 에서 빠르게 타이핑할 때 *"캐럿이 색을 바꿨다"* 로 읽히는가, *"막대가 따라온다"* 로 읽히는가. 후자면 REJECT.

착수 전 실측 2건 선행: (a) 100%/150%/200% 혼합 다중 모니터에서 unaware 앱 대상 `rcCaret`→스크린 좌표 정확도, (b) 메모장/VS Code/Chrome/Terminal 의 edit-control `rcCaret` 폭 오차.

## 후속 확인 (같은 세션, 사용자 질문 3건)

### 1. 캐럿 감지 커버리지 — 구멍은 실재하고, 일부는 기존 한계와 동근

| 앱 | 상태 | 근거 |
|---|---|---|
| Windows Terminal | 캐럿 못 잡음 | `microsoft/terminal#6805`(Priority-1, 22H1 fix-committed 이후 실측 미검증) |
| 리치에디트(riched20) | Shift+화살표/프로그램적 선택 시 `EVENT_OBJECT_LOCATIONCHANGE` 미발화 | MS 공식 문서 명시 |
| 계산기 워크시트 | 못 잡음 | 보고 확인 |
| Chromium 계열 | `ax_system_caret_win.cc` 로 가짜 시스템 캐럿 생성. **스크린리더 없이도 활성인지 미검증** | 소스 확인, 실측 안 됨 |
| **관리자 권한(High IL) 창** | 못 잡음 | **UIPI** |

**중요**: 마지막 항목은 **이 앱이 이미 겪고 있는 한계와 같은 뿌리**다. [dev-notes 2026-05-27](../dev-notes/2026-05-27-cursor-indicator.md) 확정 결론 — *asInvoker KoEnVue + admin cmd → `SendMessageTimeoutW` 즉시 ABORT*. 즉 **관리자 창에서는 지금도 한/영 감지 자체가 안 되므로**, 캐럿 트윈이 그 앱에서 새로 잃는 것은 없다.

반면 **Windows Terminal 은 다르다** — IME 상태는 Tier 1/2 로 잡히는데 캐럿만 못 잡는다. 여기가 진짜 구멍이고 위 "침묵 실패 = 거짓 신호" 위험이 실재하는 지점이다.

### 2. 커서 인디케이터 추종 부드러움 — 폴링은 이미 병목이 아니다

> **정정 (2026-07-23)** — 아래는 조사 당시(선언값 16·"≈60fps") 서술이다. `SetTimer` 실측으로 **16 요청 → 실배달 ~31ms(≈32fps)** 임이 확정됐고([dev-notes 2026-07-22](../dev-notes/2026-07-22-settimer-tick-quantization.md)), 디폴트는 **`CursorAlwaysPollMs`/`AnimationFrameMs` = 15**(실배달 ~15.6ms≈64fps) 로 적용됨. "`timeBeginPeriod(1)` 로 개선되나 전역 영향+배터리" 트레이드오프는 **실측 개선 0**이라 허구 — USER `SetTimer` 는 시스템 타이머 해상도와 별개 축. 전제 정정 요약은 [PR-28](PR-28-cursor-lag-perceptual-masking.md).

- ~~현재 `cursor_always_show=true` → **16ms(약 60fps)** 추종~~ → **현재 디폴트 15**(실배달 ~15.6ms≈64fps; [CursorOverlay.cs](../../App/UI/CursorOverlay.cs), `AnimationFrameMs=15`). 정지 검출 모드는 `CursorMotionPollMs=50`.
- 실측 모니터: **2560x1600 @ 60Hz / 3440x1440 @ 60Hz**. (조사 당시 "폴링=주사율 일치"는 16의 실배달 ~31ms 기준으로 **틀렸음** — 15 적용 후 ~15.6ms 로 60Hz 와 대략 맞춤.)
- 더 나은 추적 방식은 [dev-notes 2026-05-27](../dev-notes/2026-05-27-cursor-indicator.md) 에서 **이미 검토·거부**됨 — `WH_MOUSE_LL`(NativeAOT 콜백 위험 + 300ms OS 타임아웃 시 silent 비활성화), `WM_INPUT`(요구 대비 과대).

**진짜 병목 2가지**:

| 원인 | 설명 | 개선 가능성 |
|---|---|---|
| `SetTimer` 틱 양자화 | `uElapse` 가 ~15.625ms 격자로 양자화 — **16 요청 → 실배달 ~31ms**, **15 → ~15.6ms**. "불규칙 흔들림"이 아니라 **격자 넘김**이 주원인 | **디폴트 15 로 적용됨**. `timeBeginPeriod(1)` 은 USER `SetTimer` 에 **효과 0**(전역 영향·배터리 트레이드오프 서술은 허구) |
| **DWM 합성 지연** | 마우스 커서는 **하드웨어 커서 플레인**에서 GPU 가 직접 그림(지연 ≈ 0). 레이어드 윈도우는 DWM 합성을 거쳐 1~2프레임 뒤처짐 | **구조적 — 원리적으로 따라잡을 수 없음** |

유일하게 의미 있는 개선은 **예측 외삽**(마우스 속도로 다음 위치를 외삽해 지연 상쇄, 게임 기법). 부작용: 급격한 방향 전환 시 오버슈트 후 되돌아옴.

**착수 전 판별 필요**: 증상이 "뚝뚝 끊긴다"(타이머 흔들림)인지 "뒤처진다"(DWM 지연)인지 — **해법이 정반대**다.

### 3. 문서 불일치 발견 (별건, 미수정)

[CHANGELOG.md](../../CHANGELOG.md) 의 기능 목록에 **"캐럿 이동 감지 (마우스 클릭 재배치 시 인디케이터 표시)"** 항목이 있으나, **코드 전체에서 `GetCaretPos`·`rcCaret` 사용은 0건**이다(`GUITHREADINFO` 는 `hwndFocus` 용으로만 사용). 실제로는 포커스 변경 감지를 그렇게 표현한 것으로 보인다. 사실 확인 후 문구 정정 필요.
